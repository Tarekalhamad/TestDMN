The Promotion API makes additional offerings available to customers when criteria are met.
Update March 2022 - TMF has a new PromotionAPI version and the Promotion API is not yet aligned with it.
Promotion
Promotion can contain several patterns and they should be evaluated by their priority.
Priority >= 1 All patterns are evaluated based on priority until a pattern where all criteria are met. The remaining lower priority patterns 
are then not evaluated.
Priority = 0 If there is pattern with priority 0, that pattern should always be evaluated and the action of it should be added to the 
customer if the criteria are met.
Promotion @type and @baseType should be used in order any filtering in the clients is needed.

Promotion Pattern
"The pattern decides the conditions of promotion and the benefit of the promotion to be given to the eligible customer."
Contains the conditions (criteria groups) and benefits (action) of the promotion to an eligible party.
It should be only one action added per matched pattern, even if the MPO (addon p2p) is matched via several different criteria combinations

Criteria
Criteria parameters are linked to parent Criteria Groups.
A "criteria parameter" together with "criteria operator" and "criteria value" builds a Promotion Criteria. See below example usage.
Example usage:
Criteria
{
 "id": "Group.A.1",
 "criteriaPara":"2.6", //Selected for subscription Product Offering ID
 "criteriaOperator":"=",
 "criteriaValue":"7287" // 30 GB Product Offer ID
 @referredValueType: "ProductOffering"
},
CriteriaPara
The pattern followed when creating new criteria paras is the following (initially defined in older versions of TMF Promotion API and extended):
Configuration
Entity - Integral Part
2.* - Product Offering
3.* - Product Specification
4.* - Category
5.* - PaymentOption (Will always be 5.6 as we see it now as the PaymentOption applies on SignUp)
6.* - TradeInVendor
7.* - VoucherCampaign
8.* - VoucherCode
9.* - Communicator Operator (referredType can be ko or subnet)
12.* - Product Offering To be removed
13.* - Product Specification To be removed
14.* - Category To be removed
Flow - Fractional Part
*.1 - Product Inventory 
*.3 - Trade-in
*.4 - VoucherEvaluation
*.5 - Telenor Change
*.6 - Shopping cart
*.7 - Upgrading
*.8 - Downgrading
*.9 - Offer Change

Promotion action
Action is the offering that customer is getting after meeting all criteria in a promotion pattern. When customer is rewarded an offering via promotion, some 
relationships to any of the products/offerings from the met criteria need to be persisted in PI.
This is defined via relationshipType 
Object Description Comment
actionValue This value reflect the result of the action
actionRelations
hip
A list containing P2P relations to criteria Group. Types of relations: Addon | Discounts | Requires.
price Override amount to replace the price amount connected to the 
actionValue Addon.
This at the moment can only be used for Addons from category 
D63 - HW Discounts.
actionType 5.1 = ProductAddonOffer Only one supported at the moment.
